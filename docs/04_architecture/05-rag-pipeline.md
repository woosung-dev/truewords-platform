# RAG 파이프라인 설계 및 고도화 방향

## 기본 RAG 파이프라인 구조

```
사용자 질문
    ↓
Query Rewriting + 용어 감지 (LLM이 질문 확장/재작성)
    ↓
┌──────────┬──────────┬──────────┐
│ 말씀 DB  │ 용어사전  │ 원리강론  │  ← 별도 컬렉션
│(A,B,C,D) │ 컬렉션   │ 컬렉션   │
└────┬─────┴────┬─────┴────┬─────┘
     └──────────┼──────────┘
                ↓
         Re-ranker (Cross-encoder로 상위 10개 선별)
                ↓
         시스템 프롬프트 + 검색 결과 + 사용자 질문
                ↓
         LLM (Gemini/Claude) → 답변 + 출처 표기
```

## 3가지 조정 방법

1. **시스템 프롬프트** (가장 강력) — 답변 스타일, 용어 정의, 방어 지침
2. **참조 문서** (RAG) — 벡터 검색으로 관련 말씀 주입
3. **FAQ** — 자주 묻는 질문에 대한 사전 정의 답변

### 방향 1: 하이브리드 검색 (BM25 + 벡터) — ★★★★★

```
사용자 질문: "참부모님의 축복 의미"
    ↓
┌─────────────────────┬─────────────────────┐
│ BM25 (키워드 매칭)   │ 벡터 검색 (의미 유사도) │
│ "참부모" "축복" 정확매칭│ 의미적으로 유사한 문단  │
└────────┬────────────┴────────┬────────────┘
         └────── RRF 결합 ──────┘
              (순위 기반 융합)
                    ↓
              Top-K 결과
```

**왜 필수인가:**
- 종교 텍스트에 고유 용어 다수 → "천일국", "효정", "참부모" 등은 키워드 매칭이 벡터보다 정확
- "사랑의 의미"처럼 추상적 질문은 벡터가 강함
- RRF(Reciprocal Rank Fusion)로 양쪽 장점 활용

**Qdrant 구현:** sparse + dense vector 동일 컬렉션 저장, 동시 쿼리 가능

---

### 방향 2: 계층적 청킹 (Hierarchical Chunking) — ★★★★★

```
말씀선집 615권
    └── 권 (메타데이터: 권번호, 발행연도)
        └── 장/절 (메타데이터: 제목, 주제)
            └── 문단 (실제 검색 단위, ~300-500자)
                └── 부모 참조 (parent_chunk_id)
```

**왜 필수인가:**
- 300자 청크만으로 검색하면 문맥이 끊김
- "이 문단의 앞뒤 맥락"을 함께 전달해야 LLM이 정확한 답변 생성
- 계층 구조 → 검색 hit → 상위 절/장 맥락 자동 포함

**구현:**
```python
search_results = qdrant.search(query_vector, limit=10)
for result in search_results:
    parent_id = result.payload["parent_chunk_id"]
    parent = qdrant.retrieve(parent_id)
    context = parent.text  # 더 넓은 맥락
```

### 방향 3: Re-ranking 파이프라인 — ★★★★☆

```
1차 검색: 벡터 + BM25 → Top-50 후보
    ↓
2차 Re-ranker: Cross-encoder 모델로 정밀 재순위
    ↓
Top-10만 LLM에 전달
```

**왜 중요한가:**
- 1차 검색은 대략적 관련 문서를 넓게 가져옴
- Re-ranker는 질문과 문서를 동시에 읽고 관련도를 정밀 판단
- 60만 청크에서 정말 관련 있는 10개만 선별

**추천 모델:**
- `ms-marco-MiniLM-L-12` (가볍고 빠름)
- Cohere Rerank API (한국어 지원 양호)
- `bge-reranker-v2-m3` (다국어, 셀프호스팅)

### 방향 4: Query Expansion / Rewriting — ★★★★☆

```
원본 질문: "참부모님 축복이 뭐예요?"
    ↓ LLM 전처리
확장된 쿼리:
  - "참부모님께서 말씀하신 축복의 의미와 정의"
  - "축복식의 신학적 의의"
  - "축복 결혼의 섭리적 의미"
    ↓
3개 쿼리로 각각 검색 → 결과 병합 (dedup)
```

**왜 중요한가:**
- 사용자 질문은 대부분 짧고 모호함
- LLM이 질문을 확장하면 검색 재현율(recall) 대폭 상승
- 종교 용어를 모르는 일반 식구님 질문도 올바른 말씀에 매칭

**비용:** LLM 호출 1회 추가 (~$0.001/쿼리)

---

### 방향 5: Agentic RAG (다단계 검색-추론-검증) — ★★★☆☆

```
사용자 질문: "창조원리와 복귀원리의 관계를 설명해주세요"
    ↓
Agent Step 1: "창조원리" 검색 → 관련 말씀 5개
Agent Step 2: "복귀원리" 검색 → 관련 말씀 5개
Agent Step 3: 두 결과 비교 분석 → "관계" 추론
Agent Step 4: 추론 결과 검증할 추가 말씀 검색
    ↓
최종 답변 생성
```

**장점:** 복잡한 비교/분석 질문에 강함
**단점:** 비용 3~5배 증가, 응답 시간 5~10초
**추천:** Phase 3에서 "심층 분석" 모드로 선택적 제공

## 말씀 시간 기준 정책

### Case A: 최근 말씀 우선

- 날짜 기반 가중치 부여 (예: 최근 10년 x1.5, 20년 전 x0.8)
- Qdrant payload에 `year` 필드 → `score x recency_weight`
- **적합:** 목회 현장 가이드, 실천 지침 챗봇

### Case B: 전체 말씀 균형 (순수 의미 유사도)

- 시간 가중치 없이 순수 벡터 유사도로 검색
- 시스템 프롬프트: "최근 말씀과 이전 말씀 상충 시 최근 말씀 맥락 우선 설명 + 이전 말씀 병기"
- **적합:** 말씀 연구, 학술 활용, 원리 탐구

### 추천: B 기본값 + A 옵션

```
기본 모드: 전체 말씀 균형 검색 (순수 유사도)
    ↓
시스템 프롬프트: "상충 시 최근 말씀 맥락 우선 설명 + 이전 말씀 병기"
    ↓
UI 옵션: [✓] 최근 말씀 우선 (토글)
    → ON 시: recency_weight 적용
```

**이유:**
1. 전체 말씀 활용으로 투자 대비 가치 높음
2. 시스템 프롬프트 + UI 토글로 쉽게 추가 가능
3. 레드팀이 두 모드 비교 테스트 가능
4. 다중 챗봇 버전과 연동 ("최근 말씀 중심 챗봇" 별도 제공 가능)
