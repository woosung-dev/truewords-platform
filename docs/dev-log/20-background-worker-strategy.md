## 현재 전략 (MVP)

별도 워커 **도입하지 않음.** 아래 방법으로 충분히 대응 가능:

### 1. 시간 분리 (권장)
- 임베딩은 서비스 오픈 전 또는 트래픽 없는 시간에 실행
- 추가 인프라 불필요

### 2. 별도 스크립트 실행
- `python scripts/run_embedding.py` 등으로 서빙 서버와 분리 실행
- 서빙 서버에 영향 없음

### 3. Semaphore 제한 (동시 실행이 불가피할 때)

## 향후 도입 계획: ARQ

**도입 시점:** 사용자가 늘어서 "임베딩 중에도 서비스가 안정적이어야 한다"는 요구가 생길 때

### 왜 ARQ인가?

| 기준 | ARQ | Celery |
|------|-----|--------|
| async 호환 | 네이티브 (FastAPI와 동일) | sync 기반, 래핑 필요 |
| 인프라 | Redis만 필요 | Redis/RabbitMQ + 워커 + Beat + Flower |
| 작업 특성 | I/O-bound에 최적 | CPU-bound에 강점 |
| 운영 복잡도 | 낮음 | 높음 |
| 적합 규모 | 소~중규모 | 대규모 분산 |

임베딩 파이프라인은 **I/O-bound** (Gemini API 대기 ~80%, Qdrant 업서트 ~15%)이므로
async 네이티브인 ARQ가 리소스 효율 측면에서 유리하다.

### 전환 비용
- 기존 async 코드를 거의 그대로 ARQ 태스크로 이동 가능
- Redis 추가 구성만 필요

### 구성 예시 (도입 시)
```
FastAPI 서버  ──(태스크 등록)──▶  Redis  ◀──(폴링)──  ARQ 워커
```
- 컨테이너: FastAPI 1대 + ARQ 워커 1대 + Redis 1대
