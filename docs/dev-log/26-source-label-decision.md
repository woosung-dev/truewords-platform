# ADR-26: 데이터 Source 라벨 체계 결정

> **날짜:** 2026-04-11
> **상태:** 확정
> **관련:** `docs/dev-log/24-rrf-score-threshold-fix.md` §4-1

---

## 배경

설계 문서에서는 데이터 소스를 `A/B/C/D`로 분류했으나, 실제 적재된 데이터는 `L/M` 라벨을 사용 중 (11,922 포인트). 챗봇 설정의 source 필터와 실제 데이터 라벨이 불일치하는 문제가 있었다.

## 검토한 옵션

### 옵션 A: "라벨은 데이터가 정한다" (채택)
- 실제 적재된 라벨(L, M, 향후 추가 라벨)을 single source of truth로 사용
- 설계 문서의 A/B/C/D는 논리적 분류 예시로 격하
- SearchTierEditor에서 Qdrant의 실제 source 값을 동적으로 읽어와 표시

### 옵션 B: "설계가 정한다"
- 인제스트 파이프라인에서 A/B/C/D 라벨로 정규화
- 기존 데이터 재적재 필요 (L→A, M→B 등 매핑)

## 결정

**옵션 A 채택.**

### 근거

1. 데이터가 계속 추가될 예정이므로, 매번 매핑 규칙을 정하는 오버헤드 불필요
2. 기존 11,922개 포인트 재적재 비용 회피
3. SearchTierEditor가 이미 동적 구조 — 라벨 값에 무관하게 동작
4. Source 라벨은 사용자에게 노출되지 않는 내부 필터링용 값

## 후속 작업

- [ ] SearchTierEditor에서 Qdrant 실제 source 값을 동적으로 조회하여 버튼 표시
- [ ] 설계 문서에서 A/B/C/D를 "논리적 분류 예시"로 표기 수정
