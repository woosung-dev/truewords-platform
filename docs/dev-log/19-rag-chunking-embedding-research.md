## 0. 핵심 경고: 대부분의 "최적값"은 조건부다

Codex 팩트체크에서 6개 주요 주장 중 **0개가 완전 검증**되었다.
"최적 청크 크기", "최적 임베딩 차원" 같은 주장은 대부분:
- 특정 데이터셋(KorQuAD, MS MARCO 등)에서의 결과
- 특정 토크나이저(GPT BPE vs Gemini SentencePiece) 기준
- 영어 벤치마크의 한국어 외삽

**→ 자신의 데이터로 A/B 테스트하는 것이 유일한 정답.**

### 한국어 토큰 팽창은 토크나이저마다 다르다 [팩트 — Codex 검증]

| 토크나이저 | 한국어 1자당 토큰 수 | 500자 = 몇 토큰? | 비고 |
|-----------|-------------------|-----------------|------|
| GPT BPE (tiktoken) | ~2-3 토큰 | 1,000-1,500 | 영어 중심 학습 |
| Gemini SentencePiece | ~0.67 토큰 | ~333 | CJK 효율적 [추측] |
| BERT WordPiece | ~1.5-2 토큰 | 750-1,000 | 모델별 차이 큼 |

**시사점**: "800-1200 토큰이 한국어 최적"이라는 주장은
**어떤 토크나이저 기준인지 명시하지 않으면 의미 없다.**

### 본 프로젝트 기준: Gemini embedding-001 (SentencePiece)

```
현재 설정: 500자 ≈ 333 Gemini tokens
→ 영어 최적 512 tokens에 비해 다소 작은 범위
→ 하지만 KorQuAD 연구(200자 최적)와 비교하면 충분히 큰 편
```

## 5. 결론

**가장 정직한 답변:**

현재 500자 + 1536dim + KSS + 1문장 오버랩은 **나쁘지 않은 설정**이다.
"700-800자가 최적"이라던 내 이전 추천과, "800-1200 토큰이 최적"이라는 Desktop 추천은
모두 **한국어 벤치마크 근거가 부족**했다.

2026 한국어 RAG 연구에서는 오히려 **200자가 최적**이라는 반대 데이터가 있다.
종교 텍스트가 일반 QA와 다르다는 점을 고려하면, **400-600자가 합리적 시작점**이고
현재 500자는 이 범위 안에 있다.

**확실히 개선할 수 있는 것:**
1. **오버랩 1→2문장**: 비용 없이 문맥 보존 강화 (10-20% 범위 진입)
2. **A/B 테스트**: 실제 데이터 기반 검증이 유일한 정답
3. **Contextual Retrieval**: 청크 앞에 출처 정보 prepend

**검증 없이 변경하면 안 되는 것:**
1. 임베딩 차원 1536→768 (원본 연구 미확인)
2. 청크 크기 대폭 변경 (한국어 최적값 미정립)

## 6. 추가 확인 사항 (2026-04-09)

임베딩 모델 비교 리서치 및 실제 운영 테스트를 통해 아래 사항이 추가 확인됨.

### Gemini embedding-001 유지 근거 강화

| 항목 | 확인 내용 |
|------|---------|
| MTEB Multilingual | **1위 (68.32)** — BGE-M3 (63.0) 대비 5점+ 우위 |
| 하이브리드 검색 | Gemini dense + Qdrant/BM25 sparse + RRF 융합으로 이미 구현 완료 |
| 비용 | 유료 1등급에서도 $0.15/1M 토큰 — 61만 청크 전체 인제스트 약 $18 |
| 무료 티어 | 동적 배치(31,000자 상한)로 TPM 30K 내 안정 동작 확인 |

### BGE-M3 전환은 당분간 불필요

추천 목록에서 BGE-M3의 최대 강점으로 꼽힌 2가지:
1. **하이브리드 검색** → 이미 별도 BM25로 구현 완료
2. **파인튜닝 가능** → 검색 품질 평가 후 필요 시 재검토

파인튜닝 없이 바닐라 BGE-M3는 Gemini 대비 MTEB 5점 열세.
GPU 서버 비용(월 $30-50)도 추가 발생. 현 시점에서 전환 ROI 낮음.

### 무료 티어 운영 시 핵심 발견

- **429 원인은 TPM이었음**: 실제 청크 평균 629자(≈421토큰), 90개 배치 시 ~38K 토큰 → TPM 30K 초과
- **동적 배치로 해결**: 글자 수 합계 기반으로 배치 크기 자동 조절 (39~61청크/배치)
- **GEMINI_TIER 환경변수**: `free|paid` 한 줄로 TPM/RPM/RPD 설정 자동 전환

### 재평가 시점

- 종교 텍스트 파인튜닝이 필요하다고 판단될 때 → BGE-M3 검토
- Gemini embedding 후속 모델 출시 시 → 벤치마크 재비교
- 검색 품질 평가 데이터 30~50개 질문 세트 확보 후 → A/B 테스트
