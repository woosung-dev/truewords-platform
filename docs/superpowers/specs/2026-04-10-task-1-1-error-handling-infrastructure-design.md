## Problem Statement

현재 TrueWords 백엔드에 **통합 에러 처리 인프라가 없다.**

구체적 문제:
1. `main.py`에 exception_handler 전혀 등록 안 됨
2. `chat/router.py`의 try/except 블록은 대부분 **dead code**:
   - `RateLimitExceededError`는 `Depends(check_rate_limit)`에서 raise → 함수 본문 진입 전이라 try가 catch 못함
   - `chat_stream`의 `InputBlockedError` except도 async generator 특성상 dead code
3. 에러 응답 포맷이 라우터마다 다름 (`{"detail": ...}`, 없음, 500)
4. 검색 파이프라인(Qdrant/Gemini) 실패 시 사용자 친화적 에러 없음
5. Cache collection 시한폭탄: `main.py:31`이 실패를 swallow → 콜렉션 없을 때 모든 /chat 요청이 예외
6. request_id 없음 → 레드팀 단계에서 사용자 제보 에러를 로그와 correlate 불가

## Goals

1. **Flutter 클라이언트가 일관된 에러 포맷을 받는다** — `{error_code, message, request_id, details?}`
2. **Dead code 제거** — 실제로 작동하는 에러 처리만 남김
3. **검색 파이프라인 실패가 사용자 친화적 메시지로 전달된다** — 503 + 재시도 안내
4. **Cascade tier isolation 유지** — tier 1 실패가 전체 503으로 비화하지 않음
5. **Cache collection 실패가 서비스를 죽이지 않는다** — graceful degradation
6. **모든 에러에 request_id 포함** — 로그 correlation 가능

## Non-Goals

- Full exception class hierarchy (실제 에러 패턴 관찰 후 결정)
- StarletteHTTPException 전역 변환 (admin 호환성 유지)
- Admin 라우터(`/admin/*`, `/datasource/*`)의 에러 포맷 변경
- SSE mid-stream 에러 프레임 contract (Phase 2 Flutter 구현 시 재평가)
- ABC/Factory 추상화

## Architecture

### 요청 흐름

```
Client Request
  │
  ▼
[RequestIdMiddleware]  ← 신규
  ├── X-Request-Id 헤더 있으면 사용
  ├── 없으면 uuid.uuid4() 생성
  └── request.state.request_id 저장
  │
  ▼
[FastAPI Router]
  │
  ▼
[Depends: check_rate_limit]
  ├── 통과 → 다음
  └── raise RateLimitExceededError ────┐
  │                                      │
  ▼                                      │
[Service.process_chat]                   │
  ├── validate_input → raise InputBlockedError ──┐
  ├── embed_dense_query → raise EmbeddingFailedError ──┐
  ├── cascading_search                               │ │
  │   ├── tier 1 실패 → log + try tier 2             │ │
  │   ├── tier 2 실패 → log + try tier 3             │ │
  │   └── 모든 tier 실패 → raise SearchFailedError ──┤ │
  ├── cache 조회 (app.state.cache_available 체크)     │ │
  │   └── False면 skip (no-op)                      │ │
  └── return response                               │ │
                                                    │ │
  ┌─────────────────────────────────────────────────┘ │
  │ ┌────────────────────────────────────────────────┘
  │ │
  ▼ ▼
[Exception Handlers (main.py 등록)]
  ├── InputBlockedError    → 400 + ErrorResponse
  ├── RateLimitExceededError → 429 + ErrorResponse + Retry-After header
  ├── SearchFailedError    → 503 + ErrorResponse (generic message)
  ├── EmbeddingFailedError → 503 + ErrorResponse (generic message)
  └── Exception (catch-all) → 500 + ErrorResponse (INTERNAL_ERROR) + 상세 로깅
  │
  ▼
Client Response (모든 에러가 동일한 ErrorResponse 포맷)
```

### 핵심 원칙

1. **에러 처리는 한 곳에서** — exception_handlers.py에 중앙 집중
2. **Service 레이어는 raise만** — 에러를 삼키지 않음
3. **예외: cascade의 tier-local 실패** — 의도적으로 삼키고 다음 tier 시도
4. **예외: cache 비가용 상태** — 플래그 확인 후 silently skip (로그 1회만)

#### `src/common/middleware.py`

```python
import uuid

from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response


class RequestIdMiddleware(BaseHTTPMiddleware):
    """모든 요청에 고유 request_id 할당.

    - X-Request-Id 헤더가 있으면 그걸 사용 (분산 추적 호환)
    - 없으면 UUID v4 생성
    - request.state.request_id에 저장하여 exception_handler에서 접근 가능
    - 응답 헤더에도 포함하여 클라이언트가 로그 correlation 가능
    """

    async def dispatch(self, request: Request, call_next) -> Response:
        rid = request.headers.get("X-Request-Id") or str(uuid.uuid4())
        request.state.request_id = rid
        response = await call_next(request)
        response.headers["X-Request-Id"] = rid
        return response
```

**의존성:** `starlette.middleware.base`

#### `src/common/exception_handlers.py`

```python
"""FastAPI exception handlers.

모든 사용자 정의 예외를 ErrorResponse 포맷으로 변환.
main.py에서 app.add_exception_handler로 등록.
"""

import logging

from fastapi import Request
from fastapi.responses import JSONResponse

from src.common.schemas import ErrorResponse
from src.safety.exceptions import InputBlockedError, RateLimitExceededError
from src.search.exceptions import EmbeddingFailedError, SearchFailedError

logger = logging.getLogger(__name__)


def _get_request_id(request: Request) -> str:
    """request.state.request_id 안전 접근 (middleware 미적용 시 fallback)."""
    return getattr(request.state, "request_id", "no-request-id")


async def input_blocked_handler(request: Request, exc: InputBlockedError) -> JSONResponse:
    """Prompt Injection 등 악의적 입력 차단."""
    return JSONResponse(
        status_code=400,
        content=ErrorResponse(
            error_code="INPUT_BLOCKED",
            message=exc.reason,
            request_id=_get_request_id(request),
        ).model_dump(),
    )


async def rate_limit_handler(request: Request, exc: RateLimitExceededError) -> JSONResponse:
    """요청 빈도 제한 초과."""
    return JSONResponse(
        status_code=429,
        content=ErrorResponse(
            error_code="RATE_LIMIT_EXCEEDED",
            message=str(exc),
            request_id=_get_request_id(request),
        ).model_dump(),
        headers={"Retry-After": str(exc.retry_after)},
    )


async def search_failed_handler(request: Request, exc: SearchFailedError) -> JSONResponse:
    """모든 검색 tier 실패 (Qdrant 전체 장애 등)."""
    logger.error(
        "SearchFailedError",
        extra={"request_id": _get_request_id(request), "reason": str(exc)},
    )
    return JSONResponse(
        status_code=503,
        content=ErrorResponse(
            error_code="SEARCH_FAILED",
            message="검색 서비스에 일시적 장애가 발생했습니다. 잠시 후 다시 시도해주세요.",
            request_id=_get_request_id(request),
        ).model_dump(),
    )


async def embedding_failed_handler(
    request: Request, exc: EmbeddingFailedError
) -> JSONResponse:
    """Gemini 임베딩 생성 실패."""
    logger.error(
        "EmbeddingFailedError",
        extra={"request_id": _get_request_id(request), "reason": str(exc)},
    )
    return JSONResponse(
        status_code=503,
        content=ErrorResponse(
            error_code="EMBEDDING_FAILED",
            message="검색 준비 중 오류가 발생했습니다. 잠시 후 다시 시도해주세요.",
            request_id=_get_request_id(request),
        ).model_dump(),
    )


async def unhandled_exception_handler(request: Request, exc: Exception) -> JSONResponse:
    """예상치 못한 모든 예외에 대한 catch-all.

    상세 정보는 로그에만 기록, 응답에는 generic 메시지.
    """
    logger.exception(
        "Unhandled exception",
        extra={"request_id": _get_request_id(request)},
    )
    return JSONResponse(
        status_code=500,
        content=ErrorResponse(
            error_code="INTERNAL_ERROR",
            message="서버 내부 오류가 발생했습니다.",
            request_id=_get_request_id(request),
        ).model_dump(),
    )
```

**의존성:** `ErrorResponse`, `safety/exceptions`, `search/exceptions`

#### `src/search/cascading.py`

```python
# 기존 cascading_search 함수 내부 수정
import logging

from src.search.exceptions import SearchFailedError

logger = logging.getLogger(__name__)


async def cascading_search(
    client: AsyncQdrantClient,
    query: str,
    config: CascadingConfig,
    top_k: int = 10,
    dense_embedding: list[float] | None = None,
) -> list[SearchResult]:
    """비동기 티어별 순차 검색 with tier-level failure isolation.

    tier 단위 실패는 log + 다음 tier 시도.
    모든 tier가 실패하면 SearchFailedError raise.
    """
    dense = dense_embedding if dense_embedding is not None else await embed_dense_query(query)
    sparse = await embed_sparse_async(query)

    all_results: list[SearchResult] = []
    tier_failures = 0

    for tier_idx, tier in enumerate(config.tiers):
        try:
            results = await hybrid_search(
                client,
                query,
                top_k=top_k,
                source_filter=tier.sources,
                dense_embedding=dense,
                sparse_embedding=sparse,
            )
        except Exception as e:
            logger.warning(
                "Tier %d search failed: %s. Trying next tier.",
                tier_idx,
                e,
            )
            tier_failures += 1
            continue

        qualified = [r for r in results if r.score >= tier.score_threshold]
        all_results.extend(qualified)

        if len(all_results) >= tier.min_results:
            break

    # 모든 tier 실패 시 raise
    if tier_failures == len(config.tiers):
        raise SearchFailedError(
            f"All {len(config.tiers)} search tiers failed"
        )

    return all_results
```

```
  ├── init_db: 성공
  ├── ensure_cache_collection: 실패 (콜렉션 없음)
  │   → warning 로그
  │   → app.state.cache_available = False
  └── 앱 시작 완료

이후 POST /chat:
  ├── process_chat → cache.lookup()
  │   └── cache_available=False → return None
  ├── 검색 계속 진행 (miss로 처리)
  └── 200 OK (정상 응답)
```

## Error Handling

- **중앙 집중:** 모든 에러 응답은 `exception_handlers.py`가 생성
- **Service 레이어는 raise만:** 에러 삼키지 않음 (예외: cache graceful degradation)
- **Cascade tier-local 실패는 의도적 삼킴:** log + 다음 tier 시도
- **Cache 비가용 상태:** 플래그 확인 후 silent skip
- **SSE mid-stream 에러:** Phase 1 범위 밖 (TODOS.md에 등록됨)
- **Admin 라우터:** 기존 HTTPException `{"detail": ...}` 포맷 그대로 유지

## Implementation Order (Thin Slice)

각 단계는 **end-to-end 동작하는 상태**로 끝남. 중간에 중단되어도 partial value 있음.

**중요:** `chat/router.py`의 try/except 블록 전체는 **Slice 2 종료 시점**에 한꺼번에 제거. Slice 1에서 부분 제거하면 RateLimitExceededError가 글로벌 핸들러 없이 떠도는 중간 상태가 생김. Slice 2에서 모든 글로벌 핸들러(InputBlocked + RateLimit)가 등록된 후에만 안전하게 전체 제거 가능.

1. **Slice 1:** ErrorResponse schema + RequestIdMiddleware + InputBlockedError handler 등록
   - 가장 작은 end-to-end: ErrorResponse 모델 + middleware + 한 개 handler
   - Stage 1 테스트 전체 + Stage 2의 input_blocked_handler 테스트 + Stage 3의 InputBlockedError REGRESSION
   - **`chat/router.py`는 아직 건드리지 않음** (기존 try/except 유지)
   - 이 시점에 InputBlockedError는 **라우터 try/except가 먼저 catch**하므로 글로벌 핸들러는 아직 작동 안 함. 테스트는 글로벌 핸들러 직접 호출로 검증.

2. **Slice 2:** RateLimitExceededError handler 등록 + `chat/router.py` try/except 블록 전체 제거
   - 글로벌 핸들러 2개 등록 완료 상태에서 라우터의 dead code 일괄 제거
   - 이제 모든 예외가 글로벌 핸들러로 흐름
   - Stage 2의 rate_limit_handler 테스트 + Stage 3 전체 (REGRESSION 2건 포함)
   - **이 시점에 REGRESSION 테스트 2건이 실제로 의미를 가짐** (진짜 end-to-end 경로 검증)

3. **Slice 3:** search/exceptions.py + cascade redesign + embedding wrapping + 관련 handler 2개
   - SearchFailedError / EmbeddingFailedError 도입
   - cascade tier isolation 재설계
   - `chat/service.py`에서 embed_dense_query를 try/except로 감싸 EmbeddingFailedError로 래핑
   - search_failed_handler + embedding_failed_handler 등록
   - Stage 2의 해당 핸들러 테스트 + Stage 4 전체

4. **Slice 4:** Exception catch-all handler 등록
   - 모든 예상치 못한 예외 처리 + 상세 로깅
   - Stage 2의 unhandled_exception_handler 테스트 3건
   - `tests/test_chat_router_errors.py`의 unknown exception 시나리오

5. **Slice 5:** Cache graceful degradation
   - `main.py` lifespan 수정 → `app.state.cache_available` 플래그
   - `src/chat/dependencies.py::get_cache_service` 수정 → `None` 반환 로직
   - Stage 5 테스트

**Slice 간 의존성:**
- Slice 1 → Slice 2: RequestIdMiddleware가 있어야 REGRESSION 테스트에서 request_id 검증 가능
- Slice 2 → Slice 3: 라우터 정리 완료 후 새 예외 추가 (충돌 방지)
- Slice 3 → Slice 4: 구체 예외 핸들러가 먼저 등록되어야 catch-all이 덮어쓰지 않음
- Slice 4 → Slice 5: cache degradation은 독립적이지만 catch-all이 있으면 테스트 실패 시 debug 쉬움
