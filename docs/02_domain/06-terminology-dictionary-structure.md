# 종교 용어 사전(대사전) 구조 설계

## 핵심 질문

대사전 데이터를 말씀 본문과 **같은 레이어(컬렉션)**에 넣을지, **분리**할지.

## 5가지 방안 비교

### 방안 1: 시스템 프롬프트 내 핵심 용어 삽입 — ★★★★★

```
시스템 프롬프트:
"""
다음은 핵심 종교 용어 정의입니다. 답변 시 이 정의를 기준으로 하세요:
- 축복: 참부모님으로부터 받는 결혼 축복식...
- 천일국: 하늘 부모님 아래 하나된 가정의 나라...
- 효정: 하늘 부모님을 향한 자녀의 효심과 정성...
(100~200개 핵심 용어)
"""
```

| 항목 | 내용 |
|------|------|
| **장점** | LLM이 항상 이 정의 기준으로 답변 (가장 강력), 구현 비용 거의 없음 |
| **단점** | 200개가 한계 (토큰 비용 + 컨텍스트 윈도우) |
| **적용** | 가장 중요한 100~200개 용어 엄선 |

---

### 방안 2: 별도 컬렉션 + 쿼리 시 동적 병합 — ★★★★☆

```
Qdrant 컬렉션 구조:
├── malssum_collection (말씀 본문 청크)
├── dictionary_collection (대사전 항목)  ← 별도!
└── wonri_collection (원리강론 청크)

검색 흐름:
사용자 질문 → 용어 감지 → dictionary_collection에서 해당 용어 검색
    ↓
용어 정의 + 말씀 검색 결과 → LLM에 함께 전달
```

| 항목 | 내용 |
|------|------|
| **장점** | 대사전 전체 수천 항목 수용, 말씀 검색 오염 없음, 관련 용어만 동적 주입 |
| **단점** | 용어 감지 로직 필요, 검색 호출 1회 추가 |

**구현 예시:**
```python
# 1. 질문에서 종교 용어 감지
detected_terms = detect_religious_terms(question)

# 2. 용어 사전에서 정의 가져오기
term_definitions = qdrant.search(
    collection="dictionary_collection",
    query=question,
    limit=5
)

# 3. 말씀 본문 검색
malssum_results = qdrant.search(
    collection="malssum_collection",
    query=question,
    limit=10
)

# 4. LLM에 함께 전달
prompt = f"""
[관련 용어 정의]
{term_definitions}

[관련 말씀]
{malssum_results}

[질문]
{question}
"""
```

### 방안 3: 같은 컬렉션에 메타데이터로 구분 — ★★★☆☆

```
malssum_collection:
├── {text: "축복은...", type: "dictionary", source: "대사전"}
├── {text: "참부모님 말씀...", type: "malssum", source: "선집 제1권"}
├── {text: "효정의 정의...", type: "dictionary", source: "대사전"}
└── ...
```

| 항목 | 내용 |
|------|------|
| **장점** | 관리 포인트 1개, `filter: {type: "dictionary"}`로 분리 가능 |
| **단점 (비추 이유)** | 말씀 검색 시 사전 항목이 상위 노출 가능, 필터 누락 시 오염, 컬렉션 비대화 |

> ⚠️ **비추천**: "축복" 검색 시 대사전의 "축복 정의"가 실제 말씀보다 높은 유사도로 올라옴 → 말씀 원문 대신 사전 정의를 답변에 사용하게 됨

---

### 방안 4: 임베딩 전처리에서 용어 확장 — ★★★☆☆

```
원본 청크: "참부모님께서 축복의 의미를 말씀하셨다"
    ↓ 전처리
확장된 청크: "참부모님께서 축복의 의미를 말씀하셨다
[용어: 축복 - 참부모님으로부터 받는 결혼 축복식, 혈통전환의 의식]
[용어: 참부모 - 인류의 참된 부모, 재림주]"
```

| 항목 | 내용 |
|------|------|
| **장점** | 검색 시 용어 의미 포함으로 매칭 정확도 향상, 별도 검색 불필요 |
| **단점** | 청크 크기 증가 → 임베딩 품질 저하, 용어 변경 시 전체 재임베딩 |
| **적용** | 핵심 50~100개 용어에 대해 선별적 적용 |

---

### 방안 5: Knowledge Graph 기반 — ★★☆☆☆

```
[참부모님] --말씀하심--> [축복]
[축복] --정의--> [결혼 축복식]
[축복] --관련--> [혈통전환]
[축복] --근거--> [선집 제45권 p.123]
[혈통전환] --관련--> [타락론]
```

| 항목 | 내용 |
|------|------|
| **장점** | 용어 간 관계 표현, 복잡한 신학적 추론 가능 |
| **단점** | 구현 비용 매우 높음 (Neo4j 등), 관계 정의에 신학 전문성 필요, 수개월 소요 |
| **결론** | Phase 1에서는 과도한 투자 |

## 요약 비교

| 순위 | 방안 | 추천도 | 같은 레이어? |
|------|------|--------|------------|
| 1 | 시스템 프롬프트 내 핵심 용어 (100~200개) | ★★★★★ | 해당없음 (프롬프트) |
| 2 | 별도 컬렉션 + 동적 병합 | ★★★★☆ | ❌ 분리 |
| 3 | 같은 컬렉션 + 메타데이터 구분 | ★★★☆☆ | ⚠️ 가능하나 비추 |
| 4 | 임베딩 전처리 용어 확장 | ★★★☆☆ | 같은 레이어에 녹임 |
| 5 | Knowledge Graph | ★★☆☆☆ | 완전 별도 시스템 |

## 추천 조합: 1번 + 2번 병행

```
시스템 프롬프트
├── 핵심 용어 100~200개 (항상 포함, 해석 기준선)
│
검색 시
├── dictionary_collection에서 질문 관련 용어 3~5개 동적 주입
├── malssum_collection에서 관련 말씀 10개 검색
│
LLM에 전달
├── [핵심 용어 정의] (시스템 프롬프트)
├── [추가 관련 용어] (동적 주입)
├── [관련 말씀 본문] (검색 결과)
└── [사용자 질문]
```

## 같은 레이어에 두면 안 되는 이유

> 대사전의 "축복" 정의 문서와 아버님이 실제로 축복에 대해 말씀하신 원문의 임베딩 유사도가 높을 수밖에 없습니다. 같은 컬렉션에 넣으면 사전 정의가 원문을 밀어낼 수 있어, **"말씀 챗봇"이 아니라 "사전 챗봇"**이 되는 위험이 있습니다.
